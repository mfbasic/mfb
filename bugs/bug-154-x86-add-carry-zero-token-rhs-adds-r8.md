# bug-154 — x86_64 `add_carry`/`sub_borrow` with a zero-token `rhs` adds garbage `r8` instead of 0 (corrupts PCG64 seed on x86)

Last updated: 2026-07-12
Effort: small (<1h)
Severity: HIGH
Class: Correctness (x86-only miscompile of a core builtin)

Status: Open
Regression Test: _(none yet — see Validation)_

On linux-x86_64, the register-form `add_carry`/`sub_borrow` encoders do not treat
a zero-token (`abi::ZERO` → `"xzr"`) `rhs` as zero: they emit the ALU op against
register sentinel 16, which `alu_rr` encodes as `r8` (REX.R + `16 & 7 == 0`). The
PCG64 seed dance emits `add_carry(&hi, abi::ZERO, &hi, abi::ZERO, &carry)` to
compute `hi = hi + 0 + carry` (propagate the low-word carry into the high state
word). On x86 this becomes `adc hi, r8`, i.e. `hi = hi + r8 + carry`, folding the
caller's leftover `r8` into the RNG's high state word. The single correct
behavior: a zero-token `rhs` must contribute exactly 0, so `math::seed(n)`
produces the defined PCG64 stream (identical to aarch64/riscv) on x86.

Because the corruption is silent and the x86 acceptance goldens were captured
from this same compiler, golden diff does not catch it — it is a cross-platform
RNG divergence and a spec violation, not a golden mismatch.

References:

- PCG64 seeding: `math-rng-pcg64` memory note (arena RNG state offsets); the
  zero-token/`abi::ZERO` neutral op is from plan-00-G §4 (`explicit-carry-ops`).
- Contrast fix bug-123 (same file) already special-cased a zero-token
  `carry_out`/`borrow_out` to avoid corrupting `r8` on the setcc — the `rhs`
  input was left unguarded.
- goal-03 review.

## Failing Reproduction

```
' seed the RNG and print a few draws
math::seed(12345)
PRINT math::rand()   ' x86 differs from aarch64 for the same seed
```

- Observed (x86_64): a stream that differs from the algorithmically-defined
  PCG64 output for seed 12345 (high state word polluted by `r8`).
- Expected: the same stream every platform produces for that seed.

Contrast: aarch64 `emit_add_carry` encodes `adcs dst, lhs, xzr` where physical
register 31 reads as zero, so the same MIR is correct there. This is why the bug
is x86-only.

## Root Cause

`src/arch/x86_64/encode/emitter.rs:2064` `enc_add_carry` (and `:2104`
`enc_sub_borrow`): after resolving `rhs = reg(field(.., "rhs"))` (which returns
the zero-token sentinel 16 for `"xzr"`), neither branch checks
`is_zero_token(rhs)` before `alu_rr(0x01/0x11, dst, rhs)` (`:2076`, `:2084`).
`alu_rr` (`emitter.rs:217`) sets REX.R from `src >= 8` and encodes
`modrm(0b11, src & 7, dst)`, so sentinel 16 → REX.R + reg 0 = `r8`. The op adds
`r8` rather than nothing.

Reachable caller: `src/target/shared/code/entry_and_arena.rs:1994`
(`rng_seed_at`) and the sibling `arena_fill_seed` (`emit_seed_dance`,
`:2008+`) both emit `add_carry(&hi, abi::ZERO, &hi, abi::ZERO, &carry)`.
`enc_sub_borrow`'s zero-token-`rhs` half is currently latent (no shared caller
passes `xzr` as `sub_borrow`'s rhs), but must be fixed in lockstep.

## Goal

- On x86_64, `add_carry(dst, _, lhs, abi::ZERO, carry_in)` computes
  `dst = lhs + 0 + carry_in`; `sub_borrow` analogously subtracts 0.
- `math::seed(n)` on x86_64 yields byte-identical draws to aarch64/riscv.

### Non-goals (must NOT change)

- aarch64/riscv encodings (already correct).
- The zero-token `carry_out`/`borrow_out` handling (bug-123, already correct).
- The generated bytes for `add_carry` with a *real-register* rhs.

## Blast Radius

- `enc_add_carry` (`emitter.rs:2064`) — fixed by this bug; reachable via RNG seed.
- `enc_sub_borrow` (`emitter.rs:2104`) — same defect, in scope (latent today).
- Related latent nit (separate, LOW): neither `enc_add_carry` nor
  `enc_sub_borrow` has the `dst == rhs` aliasing guard that `alu3`
  (`emitter.rs:1288`) has — if the allocator ever colors `dst` onto `rhs`, the
  `mov dst,lhs` clobbers `rhs` before it is read. Not proven reachable; fold into
  the fix by mirroring `alu3`'s guard.

## Fix Design

In both encoders, when `is_zero_token(rhs)`: skip the `alu_rr` add/sub of `rhs`
entirely for the carry-in-zero branch, and for the carry-in-register branch use
the immediate form (`adc dst, 0` / `sbb dst, 0`, opcode `0x83 /2` and `/3`, imm8
0) so the carry/borrow still flows but nothing is added. While there, add the
`dst == rhs` guard mirroring `alu3`.

## Validation Plan

- Regression: a runtime test that seeds a fixed value and asserts the first N
  `math::rand()` draws equal the known PCG64 reference vector — must fail on x86
  pre-fix, pass after, and match aarch64 output.
- Full `scripts/test-accept.sh` on x86_64 (goldens for the seed helper will shift
  — confirm the delta is only the corrected `adc`/`sbb` bytes).

## Summary

A one-line-class omission (`is_zero_token(rhs)` guard) in the x86 carry/borrow
encoders silently pollutes the PCG64 high state word with `r8`, diverging x86 RNG
from every other target. Fix both encoders symmetrically and pin with a
cross-platform seed-vector test.
