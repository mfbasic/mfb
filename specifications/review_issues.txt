REQUIREMENT: `isNumeric` must follow numeric conversion success semantics
SPECIFICATION: standard_package.md @ §3.1 General
GROUP: stdlib
STATUS: Completed
RECOMMENDED: CODE CHANGE

Resolve by making `isNumeric` reuse the same parse validity and finite/range checks as the supported numeric conversions, returning `FALSE` for malformed, non-finite, or overflow-only text, and add runtime tests for large exponents and boundary values.

REQUIREMENT: Top-level LET and MUT declarations

The compiler AST only has `Item::Function` and `Item::Type`, and the parser only accepts top-level `SUB`, `FUNC`, `TYPE`, `UNION`, and `ENUM`, so top-level bindings are rejected and have no resolver/typecheck/IR/bytecode/native model. REQUIREMENT: Source LINK native binding declarations

The spec defines `LINK "lib" AS namespace` blocks with native resource declarations, wrapper functions, `SYMBOL`, `ABI`, `SUCCESS_ON`, `ERROR_ON`, and `RETURN_OUT`. The lexer/parser/front end do not model `LINK`; top-level parsing only accepts imports and ordinary declarations, and package native metadata is derived from bytecode/imports rather than source `LINK` declarations. Resolve by adding LINK parsing, AST/resolver/typechecker support for native namespaces and resources, ABI validation, native-link table emission, resource close integration, package import behavior, native backend symbol loading, and runtime tests against a small native library.

The spec is internally inconsistent with the compiler’s established behavior. It says `Integer / Integer` produces `Float` and that `Float`/`Fixed` do not support `MOD`, while the promotion table and current compiler/tests treat `/` like promoted numeric arithmetic, `DIV` as `Float`, and `MOD` as available across numeric promotions. Resolve by updating the spec to match the implemented/tested operator model, or alternatively make a deliberate CODE CHANGE that flips `/`, `DIV`, and `MOD` behavior and updates the existing arithmetic tests.

REQUIREMENT: Project source directory package mapping

GROUP: language

`project.md` says each source directory contributes to one package namespace and child directories use their own directory names unless explicitly mapped. `mfbasic.md` §13 says the selected `.mfb` files form one source package and directories do not create package boundaries, and the compiler implements the single-package model. Resolve by updating `project.md` to match `mfbasic.md`, removing the stale directory/package mapping and `sources[*].package` namespace behavior unless a future workspace/package-mapping feature is reintroduced.

REQUIREMENT: Non-escaping MUT lambda capture
SPECIFICATION: plan-lambda-mut.md @ §7 Recommended Direction; §11 Compiler And IR Changes Required For Option B
GROUP: language
RECOMMENDED: SPEC CHANGE
The plan recommends allowing `MUT` capture for compiler-proven non-escaping callback positions such as `forEach`, but the current compiler rejects all mutable lambda captures, and `mfbasic.md` §6/§14 explicitly says non-escaping closures are not part of v1. Resolve by either marking `plan-lambda-mut.md` as future/non-normative, or promote the feature into `mfbasic.md` and implement callback non-escape metadata, type-checking, IR/lowering support, and runtime tests for safe mutable callback capture.

REQUIREMENT: Match exhaustiveness diagnostics
SPECIFICATION: mfbasic.md @ §9 Pattern Matching
GROUP: language
RECOMMENDED: CODE CHANGE
The spec requires compile-time exhaustiveness checking: unions must cover all member types, and open types such as `Integer` or `String` require `CASE ELSE`. The current typechecker only uses exhaustiveness to decide whether a `MATCH` statement always returns; non-exhaustive matches can compile when subsequent code handles fallthrough. It also counts guarded cases as covered even though a guard may fail. Resolve by emitting a match-specific diagnostic for non-exhaustive matches regardless of surrounding flow, requiring `CASE ELSE` for open scrutinee types, and treating guarded cases as non-covering unless another unguarded case or `CASE ELSE` covers the remainder.

REQUIREMENT: Float arithmetic finite-result enforcement
SPECIFICATION: mfbasic.md @ §4.1 Primitives, Numeric edge cases
GROUP: language
RECOMMENDED: CODE CHANGE
The spec requires `Float` operations that would produce NaN or infinity to fail instead: invalid domains and division by zero with `ErrInvalidArgument`, overflow to infinity with `ErrOverflow`. Native lowering for `Float` arithmetic in `src/target/shared/code/builder_numeric.rs` emits raw AArch64 floating-point add/subtract/multiply/divide/pow results and stores the bits without a post-operation finite check. Resolve by validating every `Float` arithmetic result before exposing it, mapping NaN/domain failures to `ErrInvalidArgument` and infinity/overflow to `ErrOverflow`, and adding runtime tests for overflow and invalid floating-point arithmetic paths.

REQUIREMENT: Native heap object header contract
SPECIFICATION: mfbasic.md @ §14.3.1 Native heap value contract
GROUP: language
RECOMMENDED: SPEC CHANGE
`mfbasic.md` requires arena-backed values to use a 32-byte object header before payload bytes, but the compiler and `memory_layouts.md` use specialized compact layouts: strings start with byte length, records store field slots from offset 0, and unions store the tag at offset 0 followed by payload slots. Resolve by updating `mfbasic.md` to defer concrete runtime layouts to `specifications/memory_layouts.md`, or alternatively make a large CODE CHANGE that rewrites native value lowering, field access, union wrapping/extraction, string helpers, collection storage, and package/native ABI expectations around the 32-byte header.

REQUIREMENT: `WITH` update must produce an owned record value
SPECIFICATION: mfbasic.md @ §4.2 Records; §14.3.1 Native heap value contract
GROUP: language
RECOMMENDED: CODE CHANGE
`WITH value { ... }` is specified to create a record copy, and returned heap-backed values must not point into a callee stack frame. Native lowering in `src/target/shared/code/builder_misc.rs` builds the updated record in stack storage and returns the stack address, so a `WITH` result can dangle or be overwritten after further calls/returns. Resolve by allocating the updated record in the arena like record constructors do, copying original fields into the allocation, applying updates there, and adding runtime tests that return and reuse `WITH`-updated records across function calls.

REQUIREMENT: Package-qualified names must have exactly two parts
SPECIFICATION: mfbasic.md @ §13 Modules & Packages; §19 Grammar
GROUP: language
RECOMMENDED: CODE CHANGE
The spec requires `package::identifier` only and says nested qualifiers such as `a::b::c` are illegal, but the parser accepts repeated `::` segments for imports, identifiers, and type names. Resolution then uses only the first segment as the import root, so invalid nested package-qualified names are not rejected consistently. Resolve by making qualified-name parsing accept exactly one `::` segment, emitting a syntax diagnostic for additional segments, and adding invalid tests for imports, calls, constructors, and type annotations using `a::b::c`.

REQUIREMENT: Imported package type constructors must resolve through package qualification
SPECIFICATION: mfbasic.md @ §13 Modules & Packages
GROUP: language
RECOMMENDED: CODE CHANGE
The spec says imported package constructors are addressed as `package::identifier`, but imported `.mfp` type metadata is installed under bare type names and constructor/type lookup does not canonicalize package-qualified type names. As a result, exported package records such as `comparable::Box` fail to construct or type-check even when package metadata exports `Box`, while bare names can leak into the importing scope. Resolve by keying imported type metadata by the import binding/canonical package-qualified name, canonicalizing constructor/type/member lookups, rejecting unqualified imported type use, and adding valid package-constructor/type-annotation tests plus collision tests.

REQUIREMENT: Unary numeric negation for all numeric types
SPECIFICATION: mfbasic.md @ §11 Operators
GROUP: language
RECOMMENDED: CODE CHANGE
Unary `-` is specified as a numeric operator, and the typechecker accepts it for all numeric operands, but native lowering only implements unary negation for `Integer`. `Byte`, `Fixed`, and `Float` operands reach `native code plan does not lower unary operator '-' ... yet`, so accepted programs can fail during native code generation instead of working at runtime. Resolve by adding native lowering for `Byte`, `Fixed`, and `Float` unary negation with the language’s checked numeric semantics: reject/overflow `Byte` results outside `0..255`, detect the minimum raw `Fixed` value before negation, and ensure `Float` negation preserves finite-only results. Add runtime tests covering successful negation and overflow/error cases for each affected type.

REQUIREMENT: USING secondary close failure metadata
SPECIFICATION: mfbasic.md @ §15 Resource Management
GROUP: language
RECOMMENDED: CODE CHANGE
When a `USING` body fails and the resource close also fails, the spec requires the original body error to win while the close error is emitted as diagnostic/audit metadata associated with the failed cleanup. Native cleanup lowering preserves the original pending error, but the secondary close failure is discarded and no cleanup-failure metadata is recorded or exposed. Resolve by adding cleanup-failure metadata to bytecode/native runtime cleanup paths, preserving it through audit data, and validating with runtime tests where body failure and close failure occur together.

REQUIREMENT: Resource handles must be deterministically closed or rejected outside `USING`
SPECIFICATION: mfbasic.md @ §15 Resource Management
GROUP: language
RECOMMENDED: CODE CHANGE
The spec requires resource handles to have exactly one live owner and deterministic cleanup, with `USING` as the visible lifetime construct. The compiler only emits cleanup for `USING`; ordinary `LET f = fs::openFile(...)` / `MUT f = ...` resource bindings can compile without any guaranteed close on scope exit. Resolve by either rejecting resource-producing bindings outside `USING` and explicit consuming close flows, or by adding full lexical resource-drop analysis/lowering for all normal, return, fail, propagate, and auto-propagation exits, with missing-close and double-close diagnostics plus runtime tests.

REQUIREMENT: Inner blocks may shadow enclosing bindings
SPECIFICATION: specifications/mfbasic.md @ §5 Bindings, Scope, and Assignment
GROUP: language
RECOMMENDED: CODE CHANGE
The resolver rejects inner-block shadowing because nested scopes clone the enclosing locals map and `LET`/`MUT`, `FOR`, `FOR EACH`, and `USING` bindings report `SYMBOL_DUPLICATE_LOCAL` when their name already exists in an outer scope. Resolve by replacing flat cloned local maps with lexical scope tracking: duplicate checks should apply only within the current scope, lookup should resolve the nearest binding, outer `MUT` reassignment should still target the enclosing binding when it is not shadowed, and IR/lowering should use stable scoped symbol identities so shadowed locals do not collide. Add valid and invalid tests for shadowing in `IF`, loops, `USING`, `MATCH`, and lambdas.

REQUIREMENT: Import aliases must not conflict with other imported package names
SPECIFICATION: specifications/mfbasic.md @ §13 Modules and Packages
GROUP: language
RECOMMENDED: CODE CHANGE
Import validation rejects duplicate binding names, aliases that conflict with built-in package bindings, and aliases that conflict with visible top-level declarations, but it does not compare an alias against another import's original package name. For example, `IMPORT foo AS bar` alongside `IMPORT bar AS baz` can pass even though alias `bar` conflicts with imported package name `bar`. Resolve by tracking both import bindings/aliases and original package names during per-file import validation, then reject alias-vs-package-name collisions in either order. Add invalid import tests for those collision cases.

REQUIREMENT: Pipeline tail grammar
SPECIFICATION: mfbasic.md @ §11 Operators; §19 Grammar
GROUP: language
RECOMMENDED: CODE CHANGE
The spec restricts pipeline RHS syntax to a call tail, `ident(...)` or `package::ident(...)`, with `_` used as a pipe argument placeholder. The parser accepts any expression containing `_` and substitutes every identifier `_`, so invalid forms like `x |> _ + 1` can parse. Resolve by implementing a dedicated `pipeTail` parser that accepts only call tails and only permits `_` in pipe call arguments, then add valid and invalid pipeline grammar tests.

REQUIREMENT: Function calls with more than eight arguments
SPECIFICATION: mfbasic.md @ §6 Functions; §14.3 Function calls and returns
GROUP: language
RECOMMENDED: CODE CHANGE
The language specifies ordinary parameter lists and call semantics without an eight-argument limit, but native AArch64 lowering only maps arguments 0 through 7 to registers. `src/arch/aarch64/abi.rs` returns `stack arguments are not implemented` for argument index 8 and above, so valid source functions with nine or more parameters can pass parsing/typechecking/IR and then fail during native planning/code generation. Resolve by implementing AArch64 stack-argument passing and callee access for user functions, imported/package calls, and any runtime/native helper path that can exceed register arguments; preserve left-to-right argument evaluation and error propagation, then add runtime tests for calls with exactly 8, 9, and more mixed-type arguments.

REQUIREMENT: Managed recursion exhaustion failure
SPECIFICATION: specifications/mfbasic.md @ §6 Functions
GROUP: language
RECOMMENDED: CODE CHANGE
`mfbasic.md` requires recursion and call-stack exhaustion to fail with `ErrOutOfMemory` or a specific runtime error instead of undefined behavior. Native lowering emits ordinary direct and indirect calls for user, package, and function-value calls without managed recursion-depth or stack-budget checks, and no runtime path maps stack overflow into an MFBASIC `Error`; deep recursion can therefore exhaust the host stack and crash. Resolve by adding call-depth or stack-budget accounting around generated calls, returning `ErrOutOfMemory` (`77010001`) or a documented runtime error through the normal `Result` path before the host stack is exhausted, then add a runtime test proving deep recursion errors observably rather than crashing.

REQUIREMENT: Non-entry function declarations must include parameter parentheses
SPECIFICATION: mfbasic.md @ §6 Functions; §8.7 Program entry point; §19 Grammar
GROUP: language
RECOMMENDED: CODE CHANGE
The grammar requires `FUNC ident(...) AS type` and `SUB ident(...)`, while §8.7 only makes empty parentheses optional for zero-argument executable entry points. `src/ast.rs` accepts missing parameter lists for any `FUNC` or `SUB`, so non-entry declarations like `SUB helper` compile. Resolve by requiring `(`...`)` on all function declarations except the selected zero-argument entry point, or by explicitly broadening the spec if all zero-argument functions should allow omitted parentheses.

REQUIREMENT: Imports must appear before declarations
SPECIFICATION: mfbasic.md @ §19 Grammar
GROUP: language
RECOMMENDED: CODE CHANGE
The grammar defines `program = { import | linkDecl } { declaration }`, so imports are a leading top-level section before declarations. `src/ast.rs` accepts `IMPORT` anywhere in the top-level loop, including after function/type declarations. Resolve by tracking when the parser has consumed the first declaration and rejecting subsequent `IMPORT` statements with a parse diagnostic, while preserving leading imports before declarations.

The spec allows importing source packages and says exported templates in source packages are instantiated by the importing compilation. The resolver accepts `local:///.../project.json` and `packages/<name>/project.json` as valid source package locations, but the build pipeline later only loads `packages/<name>.mfp`; source package exports are not parsed, typechecked, monomorphized, or lowered for import. Resolve by adding source-package dependency compilation/import support, including exported symbol/type metadata, template instantiation across package boundaries, dependency ordering, cycle checks, and native/bytecode integration, or by requiring source packages to be built to `.mfp` before import and updating the specs consistently.

REQUIREMENT: MFBC section and required-flag verification
SPECIFICATION: package_format.md @ Container flags; MFB Package Bytecode; Verifier Rules
GROUP: package
RECOMMENDED: CODE CHANGE
The package format requires unknown required container flags to reject a package, required sections to be present exactly once, section ranges not to overlap, unknown required sections to reject, and optional unknown sections to be ignored only when flags permit. `src/bytecode.rs` reads section IDs into a `HashMap`, so duplicate section IDs are accepted with the last one winning; it checks only that each section is in range, not that ranges are non-overlapping; it does not read or enforce section flags; it does not enforce `GLOBAL_TABLE` even though the spec lists it as required; and the MFP reader does not reject unknown required container flag bits. Resolve by adding a real container/section verifier before package import: validate flag masks, track duplicate IDs and byte ranges, require every required section exactly once, enforce unknown-section flag semantics, require exact payload parsing, and add malformed `.mfp` tests for duplicates, overlaps, missing `GLOBAL_TABLE`, and required flags.

REQUIREMENT: Entry error exit-code mapping consistency
SPECIFICATION: package_format.md @ MANIFEST entryFlags
GROUP: package
RECOMMENDED: SPEC CHANGE
`package_format.md` says an uncaught entry error exits with `error.code`, but `mfbasic.md` §8.6/§8.7 specifies exit code `255` with `Code: <err.code> Message: <err.message>`, and native lowering implements the `255` behavior in `src/target/shared/code/mod.rs`. Resolve by updating `package_format.md` to match the language-level entry contract and current runtime behavior.

REQUIREMENT: EXPORT_TABLE must include all exported symbol kinds
SPECIFICATION: package_format.md @ EXPORT_TABLE
GROUP: package
RECOMMENDED: CODE CHANGE
The package format requires exported types, unions, enums, union member constructors, record constructors, top-level LET/MUT, and native wrappers in `EXPORT_TABLE`, but `src/bytecode.rs` only encodes exported functions/subs there and only decodes export kinds 1 through 5. Resolve by emitting and decoding every required export kind, mapping constructor targets to type/member metadata, and updating package import/type resolution and tests to validate exported constructors and type entries through `EXPORT_TABLE`.

REQUIREMENT: ABI hash input must include full public ABI identity
SPECIFICATION: package_format.md @ ABI_INDEX
GROUP: package
RECOMMENDED: CODE CHANGE
ABI v1 requires hashes to include fully qualified exported names and caller-visible public shape, including parameter names, defaults, exported type names, field/type visibility, and resource/native/public behavior. `function_sig_hash` omits the exported function name and parameter names, while `type_sig_hash` serializes type structure but not the exported type name itself. This can make distinct public symbols with identical shapes hash the same. Resolve by extending ABI serialization to include the fully qualified export name and every required caller-visible field, then regenerate ABI index fixtures and add package compatibility tests for renamed functions, renamed parameters, and same-shape renamed types.

REQUIREMENT: Default arguments must work across package boundaries
SPECIFICATION: mfbasic.md @ §6 Functions; §14.3 Function calls and returns
GROUP: package
RECOMMENDED: CODE CHANGE
Default arguments are specified as normal call behavior, evaluated at the call site after omitted trailing parameters are filled. Same-source lowering can synthesize defaults, but imported `.mfp` function metadata only preserves a `has_default` bit in type checking/IR external parameters, and no default value is available when omitted imported arguments are lowered. Package generation also stores only constant defaults, so non-constant source defaults accepted by the frontend cannot be represented as package-call defaults. Resolve by preserving default argument values or default thunks in package metadata, loading them for imported calls, lowering omitted imported arguments in declaration order, and adding package import tests for positional and named omitted defaults.

REQUIREMENT: Imported package bytecode exports must lower to executable native code
SPECIFICATION: linker.md @ §7 Package Linking
GROUP: package
RECOMMENDED: CODE CHANGE
The linker spec requires native executable builds to read installed `.mfp` package metadata, emit native functions for reachable package exports, resolve package-to-package calls, and include runtime helpers used by package bytecode. The current native package bridge in `src/target/shared/code/mod.rs` lowers only a narrow subset of package bytecode opcodes (`LOAD_CONST`, `LOAD_DEFAULT`, `COPY`/`MOVE`, `UNWRAP_RESULT`, `GENERAL_LEN`, `ADD`, `CALL_RESULT`, and `RETURN_OK`) and rejects many valid package opcodes such as control flow, comparisons, records/unions, collections, math, most general built-ins, and resource cleanup. Resolve by either interpreting or fully lowering the complete verified MFBC instruction set used by package exports, wiring helper detection for package-only built-ins, and adding package import runtime tests that execute exported package functions using control flow, records, collections, math, filesystem/IO helpers, and package-to-package calls.


=== threads ===

REQUIREMENT: Thread bounded queue semantics
SPECIFICATION: threading.md @ §9 Queue Semantics
GROUP: threads
RECOMMENDED: CODE CHANGE
Thread APIs are wired through type checking and native helpers, but the runtime queue implementation is a single-slot inbound/outbound value with count fields; `inboundLimit` and `outboundLimit` are validated but not used to allocate bounded queues of the requested capacity. Timeout behavior is implemented with busy-wait loops rather than real synchronized queue waits. Resolve by implementing per-thread bounded queues with the requested capacities, synchronized send/receive/poll semantics, correct timeout handling, cancellation behavior, and runtime execution tests.

REQUIREMENT: Thread transfer sendability
SPECIFICATION: mfbasic.md @ §16 Threads; standard_package.md @ §9 Built-in Thread Package
GROUP: threads
RECOMMENDED: CODE CHANGE
The spec says opaque resource handles, including `File`, socket handles, and `Thread`, are not sendable to threads. The type checker validates `thread::start` and `thread::send` argument shapes, but does not reject input, message, or output types that directly or transitively contain a resource or thread handle. Resolve by adding a sendability predicate for `thread::start` `In`, `Msg`, and `Out`, plus `thread::send` message values, and preserve that metadata in package/import verification so imported workers cannot smuggle non-sendable ownership across thread boundaries.

REQUIREMENT: Thread result field access
SPECIFICATION: mfbasic.md @ §4.8 Threads; §16 Threads
GROUP: threads
RECOMMENDED: CODE CHANGE
The specification requires a completed parent `Thread OF Msg TO Out` to expose `result AS Result OF Out` for field access as `t.result`, and says `thread::waitFor(t)` returns `t.result` with normal auto-unwrapping. The type checker and IR type inference recognize `Thread.result`, but bytecode and native lowering only implement member access for records, variants, map entries, enums, and `Error`; lowering `t.result` falls through as an unknown field on `Thread`. Resolve by adding lowering/runtime support for reading the stored thread result field from `Thread`, defining its blocking or completed-only semantics consistently with `thread::waitFor`, and adding valid/invalid thread result field tests that execute generated code.

REQUIREMENT: Fresh package instance for each started thread
SPECIFICATION: mfbasic.md @ §16 Threads; threading.md @ §3 Isolation; threading.md @ §7 Thread Runtime Helpers
GROUP: threads
RECOMMENDED: CODE CHANGE
The spec requires each started thread to run the imported worker in a fresh package instance with isolated package state and runtime arena state. Native `thread::start` currently stores the caller's `ARENA_STATE_REGISTER` in the thread control block, and the trampoline restores that same arena for the worker. Resolve by allocating/initializing a worker-owned package/runtime arena state for each thread start, binding imported package exports to that instance, and adding runtime tests proving two starts do not share package state or arena-owned mutable data.

REQUIREMENT: Thread handle lexical drop behavior
SPECIFICATION: mfbasic.md @ §16 Threads
GROUP: threads
RECOMMENDED: CODE CHANGE
The spec says dropping a completed `Thread` releases queued messages, and dropping a running `Thread` requests cancellation, detaches the worker, and ensures runtime reclamation. The compiler has explicit `thread::cancel`/`waitFor` helpers but no lowering for `Thread` value drop at scope exit, reassignment, return/error cleanup, or moved-from paths. Resolve by adding ownership/drop metadata and cleanup lowering for `Thread` handles, implementing detach/reclaim behavior in the runtime helpers, and adding runtime tests for completed-thread drop, running-thread drop, reassignment, and error-path cleanup.


=== typesystem ===

REQUIREMENT: Result success constructor privacy
SPECIFICATION: standard_package.md @ §1 Built-in Types; mfbasic.md @ §4.4 Result, Error, and absence
GROUP: typesystem
RECOMMENDED: CODE CHANGE
The spec says user code may construct `Error[...]`, but may not construct `Result` or `Ok` directly. `src/typecheck.rs` currently accepts `Ok[...]` as a constructor and returns `Result OF <payload>`, making the private success member source-constructible. Resolve by rejecting source-level `Ok[...]`/`Result[...]` construction while preserving compiler-internal result wrapping and `MATCH ... CASE Ok(x)` pattern support; add valid/invalid tests for `Error`, `Ok`, `Result`, and Result pattern matching.

REQUIREMENT: Package field visibility across package boundaries
SPECIFICATION: mfbasic.md @ §4.2 Records
GROUP: typesystem
RECOMMENDED: CODE CHANGE
`PACKAGE` record fields are specified as visible only to files in the declaring package, but `src/typecheck.rs` and `src/resolver.rs` treat `Visibility::Package` as visible from any file. Imported package type metadata preserves package/private/export field visibility, so this leaks package-private fields to consumers for field reads, constructors, and `WITH` updates. Resolve by tracking the declaring package identity for source and imported symbols, making `PACKAGE` visible only when the use site belongs to the same package, and adding imported-package visibility tests for field read, constructor, and update rejection.

REQUIREMENT: Imported union package layering
SPECIFICATION: mfbasic.md @ §4.3 Unions; §20.1 Package Layering Example
GROUP: typesystem
RECOMMENDED: CODE CHANGE
The spec allows a package union to `INCLUDES` another package's union and reintroduce the included member types as members of the new union, with importers constructing them through the declaring package namespace. The compiler has partial source-local support, but union expansion and bytecode payload generation only expand includes from source-local `IrType` entries; imported package union metadata is not expanded into the including union's ABI/runtime shape. Resolve by preserving imported union member metadata through type checking, IR, bytecode type payloads, package exports, matching, constructors, and native lowering, with tests for `UNION ExtraShape INCLUDES shape::Shape` across `.mfp` package boundaries.

REQUIREMENT: Union member diagnostics must use registered error codes
SPECIFICATION: error_codes.md @ Compiler Diagnostics
GROUP: typesystem
RECOMMENDED: CODE CHANGE
`mfbasic.md` requires union members to name concrete `TYPE` declarations, and `src/typecheck.rs` reports `TYPE_UNION_MEMBER_REQUIRES_TYPE` when that rule is violated. That rule is not registered in `src/rules.rs` and is not listed in `specifications/error_codes.md`, so the compiler emits it as `UNKNOWN_RULE` instead of a canonical diagnostic. Resolve by adding `TYPE_UNION_MEMBER_REQUIRES_TYPE` to `src/rules.rs` with a new `2-203-*` code, documenting it in `specifications/error_codes.md`, and adding an invalid union-member test that verifies the registered diagnostic.

REQUIREMENT: Record comparability must be field-wise
SPECIFICATION: mfbasic.md @ §4.11 Comparable Types
GROUP: typesystem
RECOMMENDED: CODE CHANGE
Records whose fields are all comparable are specified as comparable, so `=` / `<>`, map keys, and list helpers such as `find`, `contains`, and `replace` must compare record values by their comparable fields. The type checker accepts comparable records, but native lowering falls back to comparing record value pointers for ordinary equality and uses raw inline bytes for collection/map matching, which makes records containing heap-backed comparable fields such as `String` compare by address or stored pointer bytes rather than semantic field values. Resolve by adding recursive field-wise comparison lowering for comparable records everywhere comparability is used: equality operators, list search/contains/replace, map key lookup/update/remove, and package/native bytecode paths. Add runtime tests for records with primitive fields and records with `String` fields, including use as `Map` keys.

=== filesystem ===

REQUIREMENT: Narrow `fs::openFile` mode set
SPECIFICATION: standard_package.md @ §8 Built-in Filesystem Package
GROUP: filesystem
RECOMMENDED: CODE CHANGE
`fs::openFile` and `fs::openFileNoFollow` are specified with modes `"read"`, `"write"`, and `"append"`, while `fs::open` has the wider alias set `"read"`/`"r"`, `"write"`/`"w"`, `"readWrite"`/`"rw"`, and `"append"`/`"a"`. REQUIREMENT: `fs::createTempFile()` no-argument form must work from package bytecode
SPECIFICATION: standard_package.md @ §8 Built-in Filesystem Package
GROUP: filesystem
RECOMMENDED: CODE CHANGE
The spec defines `fs::createTempFile()` with no directory argument, using the OS temp directory. Native source NIR lowering injects `fs.tempDirectory()` for this form, but bytecode lowering in `src/builtins/fs.rs` emits `OPCODE_FS_CREATE_TEMP_FILE` with no directory operand. Package/native helper metadata expects a directory string, so package bytecode that calls the no-argument form cannot execute with the specified behavior.
