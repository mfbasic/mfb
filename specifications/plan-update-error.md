# Plan: read-only `Error` with compiler-generated source location

Status: proposed (planning only - no spec or compiler changes yet)
Owner: Justin
Date: 2026-06-20

This document plans a breaking update to MFBASIC's error payload. There is no
backwards-compatibility requirement for existing `Error[...]` construction.

It complements:

- `specifications/mfbasic.md`
- `specifications/standard_package.md`
- `specifications/package_format.md`
- `specifications/memory_layouts.md`
- `specifications/architecture.md`
- `specifications/error_codes.md`

## 1. Goal

Make every user-visible `Error` carry the source location where the error
originated:

```basic
TYPE ErrorLoc
  filename AS String
  line     AS Integer
  char     AS Integer
END TYPE

TYPE Error
  code    AS Integer
  message AS String
  source  AS ErrorLoc
END TYPE
```

`ErrorLoc` and `Error` are compiler/runtime-generated read-only types. User
programs may inspect their fields, but may not construct or update either type.

User-authored errors are created through a built-in function:

```basic
FUNC error(code AS Integer, message AS String) AS Error
```

The source location of a user-authored error is the location of the `error(...)`
call expression.

## 2. Non-Goals

This plan does not add:

- user construction of `Error` or `ErrorLoc`
- user mutation or `WITH` update of `Error` or `ErrorLoc`
- optional or absent error locations
- stack traces
- catch-site rewriting of propagated errors
- compatibility for `Error[code, message]`

`Error.source` records origination, not the current propagation point.

## 3. Source Semantics

### Read-only built-in records

`ErrorLoc` has three exported fields:

- `filename AS String`
- `line AS Integer`
- `char AS Integer`

`Error` has three exported fields:

- `code AS Integer`
- `message AS String`
- `source AS ErrorLoc`

Both are read-only compiler-owned record shapes:

- field access is allowed
- construction is rejected
- `WITH` update is rejected
- assignment to fields remains impossible under the normal field model
- ordinary value binding and propagation are allowed

Example:

```basic
FUNC load(path AS String) AS String
  IF path = "" THEN
    FAIL error(77050002, "path is empty")
  END IF
  RETURN fs::readText(path)
END FUNC

FUNC main() AS Integer
  LET text = load("") TRAP(e)
    io::print(e.message)
    io::print(e.source.filename & ":" & toString(e.source.line))
    RECOVER ""
  END TRAP
  RETURN 0
END FUNC
```

### The `error` built-in

`error(code AS Integer, message AS String) AS Error` is a language built-in, not
a standard-package function. It is always in scope like `toString`.

Rules:

- `code` must be `Integer`
- `message` must be `String`
- the returned `Error.source` is the source location of the `error(...)`
  expression
- `error(...)` itself does not fail
- `FAIL error(...)` fails with that generated `Error`
- storing or returning the generated `Error` preserves its source

### Runtime-generated errors

Runtime-generated errors use the source location of the source expression whose
evaluation produced the failure.

Examples:

- `1 / x` uses the location of the division expression when `x` is zero.
- `fs::readText(path)` uses the location of the call expression when the helper
  reports a filesystem error.
- arithmetic overflow uses the location of the arithmetic expression.

When a function propagates an error, the `source` is not replaced. The original
callee/source expression remains the origin.

### Imported package errors

Errors originating inside an imported package use the package's own source
location metadata, not the caller's call site.

Errors generated by a runtime helper called directly at an importing call site
use the importer call expression's location.

## 4. Breaking Changes

Remove user construction of `Error`:

```basic
FAIL Error[77050002, "negative"]
```

becomes:

```basic
FAIL error(77050002, "negative")
```

There is no compatibility alias. `Error[...]` is rejected with a targeted
diagnostic.

Recommended diagnostics:

- `TYPE_READ_ONLY_RECORD_CONSTRUCTOR` for direct `Error[...]` and `ErrorLoc[...]`
  construction, or a more specific new rule such as `TYPE_ERROR_CONSTRUCTOR_REMOVED`
- `TYPE_READ_ONLY_RECORD_UPDATE` for `WITH` update attempts
- `TYPE_UNKNOWN_FIELD` remains appropriate for invalid field names

If new diagnostic rules are added, update:

- `specifications/error_codes.md`
- `specifications/mfbasic.md`
- `specifications/standard_package.md`
- `src/rules.rs`

## 5. Front-End Implementation

### Lexer and parser

Tokens already carry line and character positions. Preserve enough source
location to attach a span to every expression and statement that can originate a
runtime error.

Add an internal source location shape, for example:

```rust
struct SourceLoc {
    file: String,
    line: usize,
    char: usize,
}
```

The exact Rust type may differ, but it must be serializable through the compiler
pipeline and package format.

Parser requirements:

- every `Expression` has a source location
- every `Statement` keeps its existing line and gains character/file data where
  needed
- generated AST JSON may include source locations, or they may remain internal,
  but acceptance goldens must be updated consistently

### Resolver

Update built-in names:

- reserve `Error`
- reserve `ErrorLoc`
- reserve `error`

`Error` and `ErrorLoc` remain type names. `error` resolves as a built-in
function, not a package-qualified symbol.

### Type checker

Add type rules:

- `ErrorLoc` parses as a built-in type
- `Error` remains a built-in type
- `error(Integer, String) AS Error`
- `Error[...]` is rejected
- `ErrorLoc[...]` is rejected
- field access on `Error.source` returns `ErrorLoc`
- field access on `ErrorLoc.filename` returns `String`
- field access on `ErrorLoc.line` returns `Integer`
- field access on `ErrorLoc.char` returns `Integer`
- `WITH` update rejects both types as read-only

`FAIL` continues to require an expression of type `Error`.

## 6. IR and NIR

The current IR/NIR does not carry source locations on operations or values.
Add source location metadata to every IR/NIR node that can generate a runtime
error.

Minimum required sites:

- built-in and user function calls
- `error(...)`
- arithmetic operations
- numeric conversions
- string and collection indexing/slicing/search operations that can fail
- resource operations that can fail
- thread operations that can fail
- explicit `FAIL`
- `PROPAGATE` and auto-propagation edges only insofar as they preserve existing
  error values

Recommended representation:

```text
IrValue::Call { target, args, source }
IrValue::CallResult { target, args, source }
IrValue::BuiltinError { code, message, source }   ' optional lowering form
IrOp::Fail { error, source }
```

The exact enum names may differ. The invariant is that code generation can
recover the correct origin for any newly created runtime error without guessing.

Propagation must copy/forward the existing `Error` unchanged.

## 7. Runtime and Native ABI

### Value layout

Update allocated `Error` from two words to three words:

```text
Error
  +0  Integer code
  +8  String message
  +16 ErrorLoc source

ErrorLoc
  +0  String filename
  +8  Integer line
  +16 Integer char
```

All copy, thread-transfer, trap-materialization, result-error extraction, and
cleanup paths must treat `Error` as a three-field record.

### Result registers

Native fallible calls currently return an error as tag/code/message. Extend the
error-result ABI to include source location, either as:

- a fourth result register containing an `ErrorLoc` pointer, or
- a pointer to a complete `Error` object for the error case

The register approach is less disruptive to success results but requires every
error-emitting helper to also produce/source an `ErrorLoc`. The object approach
may simplify propagation but changes more call paths.

Whichever representation is chosen, the invariant is:

- success results are unchanged semantically
- error results always carry code, message, and source
- trapping materializes an `Error` with all three fields
- direct propagation preserves source without replacement

### Runtime helper source locations

Runtime helpers need a way to receive or access the source location of the
call/check they are serving.

Acceptable strategies:

- pass a compile-time source-location constant pointer as an extra hidden
  argument to fallible helpers
- emit helper-specific wrapper code at each call site that stamps returned
  code/message with the call-site location
- maintain a per-function source-location table and pass an index

Do not use a default, empty, or unknown location for implemented behavior. If a
runtime path can create an `Error`, it must create a real `ErrorLoc`.

## 8. Package and Binary Representation

Update reserved built-in type IDs:

- keep `Error` as its existing reserved ID if possible
- add a reserved ID for `ErrorLoc`
- move the first user/table type ID if needed

Update package metadata to preserve source locations for imported package code.
At minimum, package binary representation must carry enough source map data for
runtime errors originating inside package functions to materialize:

- package/source filename
- line
- char

If source files are not embedded, filenames must still be stable enough to be
meaningful in diagnostics and runtime output. Prefer package-relative source
paths over absolute build-machine paths.

Update:

- `specifications/package_format.md`
- binary reader/writer
- package verifier
- package info output if it reports built-in types or source-map metadata

## 9. Standard Library and Built-ins

Update all source standard packages and built-in generated source to use:

```basic
error(code, message)
```

instead of:

```basic
Error[code, message]
```

Built-in packages that fail from MFBASIC source, such as the JSON package, get
source locations from their own package source.

Native helpers must generate `ErrorLoc` from the source expression that invoked
the helper.

## 11. Tests and Validation

Add function tests under both required directories for the modified built-in:

- `tests/func_general_error_valid/**`
- `tests/func_general_error_invalid/**`

Valid coverage:

- `error(Integer, String)` produces an `Error`
- `FAIL error(...)` can be trapped
- trapped error exposes `code`, `message`, `source.filename`, `source.line`,
  and `source.char`
- propagated errors preserve original source
- runtime helper errors expose the call-site source
- imported package errors expose package source
- thread-transferred errors preserve source

Invalid coverage:

- wrong argument count to `error`
- wrong code type
- wrong message type
- `Error[...]` construction rejected
- `ErrorLoc[...]` construction rejected
- `WITH` update of `Error` rejected
- `WITH` update of `ErrorLoc` rejected
- unknown fields rejected on both types

Acceptance:

```sh
scripts/test-accept.sh target/debug/mfb target/accept-actual
```

Runtime validation is mandatory. At least one test must execute a generated
program and assert observable output containing the expected source filename,
line, and char.

## 12. Suggested Implementation Order

1. Add `ErrorLoc` to the type system and docs as a read-only built-in record.
2. Make `Error` read-only and add its `source AS ErrorLoc` field.
3. Add the `error(Integer, String) AS Error` built-in and reject `Error[...]`.
4. Add source locations to AST, then IR, then NIR.
5. Update native `Error`/`ErrorLoc` layouts and copy/materialization helpers.
6. Extend the result error ABI to carry source.
7. Stamp user `error(...)` calls and runtime helper failures with source.
8. Update package binary representation/source maps for imported package errors.
9. Update standard packages and all tests/goldens.
10. Run acceptance and runtime validation.

## 13. Completion Gate

This plan is not implemented until all error-producing paths create real
`Error.source` values at runtime.

Compilation success, AST/IR/NIR output, binary representation generation, or
native binary generation is not sufficient. Completion requires an execution
test proving source filename, line, and char for:

- user-authored `FAIL error(...)`
- a runtime-generated built-in/helper error
- a propagated callee error

Any remaining default, empty, unknown, zeroed, placeholder, or unsupported
source location means the feature is incomplete.
