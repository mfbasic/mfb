# Linux x86-64

The `linux-x86_64` backend writes ELF64 x86-64 executables directly, with no host
linker (`ld`, `gold`, `lld`, `gcc`, `clang`). It **shares one ELF linker** with the
aarch64 Linux backend, parameterized by an `arch` string (`"x86_64"` versus
aarch64). Only the target front-end and the ISA encoder are separate.
[[src/os/linux/link/]] [[src/target/linux_x86_64/]] [[src/arch/x86_64/]] [[src/os/linux/link/elf.rs:encode_dynamic_elf]]

A console build emits two flavors, one per dynamic loader / library naming, both
inside the project's `build/` directory:

```text
build/<project>-glibc.out
build/<project>-musl.out
```

An app-mode build (`mfb build --app`) emits one AppImage per libc world —
`build/<project>-glibc.AppImage` and `build/<project>-musl.AppImage` — like
aarch64. The two-flavor emission loop lives in the target front-end, not the
linker. [[src/target/linux_x86_64/mod.rs:write_executable]]

## Container layout

Constants are shared with aarch64: image base `0x400000`, text file offset
`0x1000`, page size `0x1000` (4 KiB). The ELF header is `ET_EXEC`, **`EM_X86_64`
(machine 62)** (aarch64 is 183), entry = `text_vmaddr + entry_offset`. There is no
`ET_DYN`/PIE path. [[src/os/linux/link/mod.rs:IMAGE_BASE]] [[src/os/linux/link/elf.rs:encode_dynamic_elf]]

A dynamic image (imports present, `encode_dynamic_elf`) has the **same program
headers** as aarch64 — seven, or eight with a read-only constant partition (see
`linux-aarch64` for the list). Only the machine field and the interpreter string
branch on arch.

The **static** image carries **two** `PT_LOAD`s — a text segment (R+X) and a
separate page-aligned **writable** data segment (R+W) — because the program entry
writes `_mfb_rt_main_arena` and the data must be writable. Both static ELF
encoders (aarch64/riscv and x86) split text and data this way, and both add
`PT_GNU_STACK` and the `PT_NOTE` provenance marker (see `provenance-marker`) for
four headers total. [[src/os/linux/link/elf.rs:encode_static_elf_x86]] [[src/os/linux/link/elf.rs:encode_static_elf]]

## Dynamic metadata

For a dynamic image the linker builds `.dynstr`, `.dynsym` (entry 0 is the null
symbol), a SysV `.hash` table, `.rela`, and `.got`, then a `.dynamic` section. The
dynamic tags are **identical to aarch64**:

```text
DT_NEEDED (one per distinct imported library)
DT_HASH DT_STRTAB DT_SYMTAB DT_STRSZ DT_SYMENT
DT_PLTGOT DT_RELA DT_RELASZ DT_RELAENT DT_PLTREL DT_JMPREL DT_PLTRELSZ
DT_FLAGS_1 = 8 (DF_1_NODELETE)
DT_NULL
```

The only arch-specific values are the relocation-type constants:
**`R_X86_64_GLOB_DAT` (6)** and **`R_X86_64_JUMP_SLOT` (7)** (aarch64 uses 1025 and
1026). Each imported function gets a **12-byte** import stub (`jmp *disp32(%rip)`
through its GOT slot, padded with `int3` to a fixed slot so the layout math is
arch-independent) and an 8-byte GOT slot. [[src/os/linux/link/mod.rs:R_X86_64_JUMP_SLOT]] [[src/os/linux/link/mod.rs:emit_import_stub]]

Symbol versioning (`.gnu.version`/`.gnu.version_r`) is emitted the same way as
aarch64 when an import carries a `version`; the encode path emits all imports
unversioned. [[src/os/linux/link/elf.rs:encode_dynamic_elf]]

## Relocations

x86-64 uses **RIP-relative rel32 in a single instruction** instead of aarch64's
`adrp; add` page pair, so it emits exactly **one relocation per reference** (it
consumes the `*Lo` relocation intent; `*Hi` is never emitted). The neutral
`RelocIntent` → x86 kind mapping is: [[src/arch/x86_64/reloc.rs:reloc_kind]]

```text
Call        -> call_pc32   (call rel32,          R_X86_64_PLT32/PC32)
DataAddr    -> data_pc32   (lea reg,[rip+disp32], R_X86_64_PC32)
GotLoad     -> got_pc32    (mov reg,[rip+disp32] GOT, R_X86_64_GOTPCREL)
```

The encoder tags each relocation with a `binding`: a `call`/data reference to a
defined symbol is `internal`/`data`; to an imported symbol it is `external`. The
linker's patch pass computes `rel = target − (site + 4)` as an `i32` and rewrites
the disp32 field, resolving an external `call_pc32` to the import's **PLT stub** and
an external `data_pc32` to the import's **GOT slot**. [[src/arch/x86_64/encode/emitter.rs:encode_instruction]] [[src/os/linux/link/mod.rs:patch_relocations]]

> Imported-data loads use the `got_pc32` relocation kind — the canonical GOT
> kind, emitted by the encoder — which the external patch arm binds by pointing
> the rel32 at the symbol's GOT slot. `data_pc32` is accepted as a historical
> alias for the same arm. [[src/os/linux/link/mod.rs:patch_relocations]]

## glibc flavor

```text
interpreter  /lib64/ld-linux-x86-64.so.2
libc.so.6        C/POSIX runtime functions (and pthread_create for thread::start)
```

Note the interpreter lives in `/lib64/` on x86 glibc (aarch64 uses `/lib/`). Each
soname an import names becomes a `DT_NEEDED` entry; the per-call `(library, symbol)`
mapping is owned by `runtime_imports` and canonical in
`./mfb spec linker import-selection`. Imported symbols use plain ELF names with no
leading underscore. `libm.so` is **not** needed — every `math::` transcendental,
`pow`, `atan2`, `tan`, and `Float MOD` lowers to an in-tree kernel. [[src/target/linux_x86_64/plan.rs:runtime_imports]]

## musl flavor

```text
interpreter  /lib/ld-musl-x86_64.so.1
libc.musl-x86_64.so.1   C/POSIX runtime functions and pthread_create
```

musl exposes the pthread entry points from libc, so `pthread_create` is imported
from `libc.musl-x86_64.so.1` rather than a separate pthread library. As on glibc,
`libm.so` is not needed. [[src/target/linux_x86_64/plan.rs:runtime_imports]]

## Entropy and app mode

The program-entry RNG seed draws from the **`getrandom` syscall (nr 318)** on
x86-64, rather than the `getentropy` libc import aarch64 uses; the `math::rand`/
`math::seed` runtime helpers and `crypto::randomBytes` still import `getentropy` on
all targets. [[src/target/linux_x86_64/code.rs:emit_random_bytes]]

App mode (GTK) shares the Linux GTK backend and builds for both libc worlds. Its
`__libc_start_main` bootstrap trampoline is per-ISA: x86 passes the trampoline's
7th argument (`stack_end`) on the stack, per System V. musl provides
`__libc_start_main` as a compat symbol, so the same trampoline serves both.
[[src/target/linux_gtk]]
[[src/target/linux_gtk/mod.rs:app_mode_imports]]

## Object plan and failure rules

`NativeObjectPlan::validate` accepts `"linux-x86_64"` alongside `"linux-aarch64"` —
the `.nobj` object/ELF plan is ISA-neutral, so the x86-64 backend reuses it
verbatim. [[src/os/linux/object.rs:validate]]

- An unsupported `(binding, kind)` relocation pair is
  `linux linker does not support relocation … …`.
- An external symbol with no stub is
  `linux-x86_64 linker cannot bind external symbol '…' from …`; the data variant is
  `linux-x86_64 linker cannot bind external data symbol '…' from …`.
- A build mode other than console or Linux app (e.g. macOS app) is rejected:
  `Linux x86-64 native targets do not support the … build mode`.

## Executable signing metadata

When the build supplies executable signing metadata, the linker emits it as a
`.mfb_sign` ELF section (shared with aarch64). Unlike macOS, Linux executables are
not otherwise signed by the linker.
