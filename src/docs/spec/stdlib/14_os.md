# OS Environment & Introspection (os)

The `os` package is the program's window onto the host process: it reads, tests,
sets, unsets, and enumerates environment variables, and reports read-only facts
about the running process and the build platform. Every call is a thin
compiler-owned runtime helper wrapping a libc primitive (or, for `os::name` /
`os::arch`, a compile-time constant baked in with no host call at all). There is
no MFBASIC source package behind `os` and no resource handle — every result is
an ordinary owned value. `os` is built in, so `IMPORT os` needs no manifest
dependency. [[src/builtins/os.rs:is_os_call]]

This topic specifies the **model** — the read/mutate/snapshot semantics of the
environment, the argument-vector and platform-constant rules of the
introspection calls, the OS seam each maps onto, and the error and concurrency
model. The per-function signatures, overloads, and examples are owned by
`./mfb man os`.

## Type and marshalling model

Every argument and every result is a plain value: environment names and values
are UTF-8 `String`s, presence is a `Boolean`, `os::pid`/`os::cpuCount` are
`Integer`, the environment snapshot is a `Map OF String TO String`, and the
argument vector is a `List OF String`. [[src/builtins/os.rs:call_return_type_name]]

Because the host C library requires NUL-terminated strings, each `String`
argument is copied into a fresh NUL-terminated arena buffer before the libc call,
and each `String` result is copied back out of the host's buffer into a fresh
owned arena `String`. This copy-in / copy-out is why a variable **name** must be
non-empty and free of an embedded NUL byte, and — for a name — free of `=`, since
the host uses `=` to separate a name from its value. A **value** must likewise be
free of embedded NUL. The `os::getEnvOr` fallback is copied by its stored byte
length rather than by NUL scan, so an embedded NUL there is preserved verbatim.
[[src/target/shared/code/os/mod.rs:marshal_cstring]]

## Environment: read, mutate, snapshot

The environment is process-global mutable state inherited from the host at
startup. `os` exposes it through three kinds of operation.

### Reads (getEnv, getEnvOr, hasEnv)

All three observe the **live** environment: they reflect both variables inherited
from the host and any change an earlier `os::setEnv` / `os::unsetEnv` made in the
same process. Each reduces to the host `getenv`; they differ only in how they
report absence.

| Call | Result | On a missing variable |
| --- | --- | --- |
| `os::getEnv` | `String` | **raises** `ErrNotFound` |
| `os::getEnvOr` | `String` | returns the caller's `fallback` (never raises) |
| `os::hasEnv` | `Boolean` | returns `FALSE` (never raises) |

The split exists so a program can decide whether absence is an error.
Critically, a variable **set to the empty string counts as present**: `getenv`
returns a non-NULL pointer to an empty string, so `os::hasEnv` reports `TRUE` and
`os::getEnv` returns `""` rather than raising. Only `os::getEnv` and `os::hasEnv`
can distinguish "set to empty" from "unset"; through `os::getEnvOr` an unset
variable and one set to `""` are indistinguishable, because both funnel to the
returned/`fallback` string. [[src/target/shared/code/os/env.rs:lower_get_env]] [[src/target/shared/code/os/env.rs:lower_has_env]]

### Mutations (setEnv, unsetEnv)

Both are SUBs returning `Nothing`; both mutate the process-global environment and
are visible to every later read in the same process and inherited by child
processes spawned afterward.

* `os::setEnv(name, value)` maps to `setenv(name, value, 1)` — it **overwrites**
  any existing value. The host rejects an empty name or a name containing `=`
  (`errno = EINVAL`); the helper distinguishes that from an out-of-memory failure
  by reading `errno`, raising `ErrInvalidArgument` for the former and
  `ErrOutOfMemory` for `ENOMEM`. [[src/target/shared/code/os/env.rs:lower_set_env]]
* `os::unsetEnv(name)` maps to `unsetenv(name)` and is **idempotent**: removing a
  variable that is not set is a no-op, not an error, so any return is treated as
  success. Afterward `os::hasEnv(name)` is `FALSE` and `os::getEnv(name)` raises
  `ErrNotFound`. [[src/target/shared/code/os/env.rs:lower_unset_env]]

### Snapshot (environ)

`os::environ()` returns a `Map OF String TO String` built by walking the host
environment array (the `NAME=VALUE` C-string vector) in two passes: one to size
the arena map, one to fill it. Each entry splits at its **first** `=` — the text
before it is the key and everything after it, *including any further `=`*, is the
value (so a `PATH`-like or `LS_COLORS`-like value keeps its embedded `=`). An
entry with no `=` at all maps its whole text to an **empty-string** value. On the
rare host that lists a name twice, the map keeps one entry for that key.
[[src/target/shared/code/os/env.rs:lower_environ]]

The returned map is an ordinary owned value **captured at the moment of the
call** — a snapshot, not a live view. A later `os::setEnv` / `os::unsetEnv` does
not change an already-returned map; re-read with a fresh `os::environ()` to
observe subsequent mutation. Like any `Map`, it is unordered.

### Concurrency

`getenv`/`setenv`/`unsetenv` are the classic non-reentrant trio: a reader must
not walk the environment array while a writer relocates or frees it. The runtime
therefore serializes **all** `os` environment (and passwd) access through one
process-global mutex. The readers (`os::getEnv`, `os::getEnvOr`, `os::hasEnv`,
`os::environ`, `os::userName`) hold it across both the libc call and the
marshal-into-arena; the writers (`os::setEnv`, `os::unsetEnv`) hold it across the
mutation — so an `os` reader in one `thread::` worker never observes a
half-relocated environment while another worker's `os::setEnv` runs. A
single-threaded program pays only one uncontended lock/unlock per call. The lock
covers `os` calls among themselves; it cannot guard a raw `getenv`/`setenv`
reached through a native `LINK` outside the package. [[src/target/shared/code/os/mod.rs:OS_ENV_LOCK_SYMBOL]]

## Process & platform introspection

The remaining calls are all nullary and read-only. They fall into three groups by
where their answer comes from.

### Build-target constants (name, arch)

`os::name` and `os::arch` are **compile-time constants** selected by the build
target — the binary is built for exactly one target, so each is materialized
directly into a fresh owned `String` with no host call and cannot fail.

| Call | Value set | Source |
| --- | --- | --- |
| `os::name` | `"macos"` or `"linux"` (the target family) | target triple prefix |
| `os::arch` | `"aarch64"`, `"x86_64"`, or `"riscv64"` | target triple suffix |

Because they are fixed per build, both are stable across every run of the same
binary. [[src/target/shared/code/os/mod.rs:os_family]] [[src/target/shared/code/os/mod.rs:os_arch]]

### Never-failing host facts (pid, cpuCount)

* `os::pid()` returns the process id as an `Integer` via `getpid` — a positive
  value, stable for the life of the process, identical on every call within one
  process. It cannot fail. [[src/target/shared/code/os/introspect.rs:lower_pid]]
* `os::cpuCount()` returns the number of online logical CPUs via
  `sysconf(_SC_NPROCESSORS_ONLN)`, **clamped to a minimum of 1** so a caller
  always gets a usable count even when the host returns `-1`/`0` for an
  indeterminate answer. It cannot fail; use it to size a `thread::` worker pool.
  [[src/target/shared/code/os/introspect.rs:lower_cpu_count]]

### The argument vector (args)

`os::args()` returns the program's command-line arguments as a `List OF String`,
**excluding the program name**: element 0 is the first *real* argument, not the
executable. A program invoked with no arguments returns an empty list. The
arguments are captured once at program startup into two runtime globals the entry
code fills before any user code runs, so `os::args` reflects the invocation
regardless of where it is called; each element is copied out into an owned
`String`. (The program name itself is available through `os::executablePath`.)
[[src/target/shared/code/os/introspect.rs:lower_args]] [[src/target/shared/code/os/mod.rs:OS_ARGV_GLOBAL_SYMBOL]]

### May-fail host lookups (hostName, userName, executablePath)

Three `String`-returning calls query host state that can be unavailable, and each
raises `ErrUnsupported` (`77050007`) when the lookup cannot produce a value.

| Call | Host mechanism | `ErrUnsupported` when |
| --- | --- | --- |
| `os::hostName` | `gethostname` into a fixed buffer | the host cannot supply its name |
| `os::userName` | `getpwuid(getuid())->pw_name` | the effective uid has no passwd entry (e.g. a bare container uid) |
| `os::executablePath` | macOS `_NSGetExecutablePath`; Linux `readlink("/proc/self/exe")` | the host cannot determine the path |

`os::userName` resolves through the passwd database rather than the controlling
terminal, so it works without a login session (for example under a service
manager); its lookup shares the same process-global lock as the environment
readers, because `getpwuid` returns a pointer into a static buffer a concurrent
lookup could overwrite mid-copy. `os::executablePath` on Linux uses the byte
count `readlink` returns (that call does not NUL-terminate) and yields the
absolute, symlink-resolved path. [[src/target/shared/code/os/introspect.rs:lower_host_name]] [[src/target/shared/code/os/introspect.rs:lower_user_name]] [[src/target/shared/code/os/paths.rs:lower_executable_path]]

### Build resources (resourcePath)

`os::resourcePath(relative As String) As String` returns the **absolute** on-disk
path of a resource the build copied out of the project's manifest `resources`
section (see `./mfb spec tooling project-manifest`). The same call resolves
correctly for every build shape because the base directory is derived at runtime
from the running executable's own path and a build-mode offset baked into the
binary:

| Build | Executable path | Resource base |
| --- | --- | --- |
| console | `…/build/<name>` | the executable's directory (`…/build`) |
| macOS `--app` | `…/Contents/MacOS/<name>` | `…/Contents/Resources` |
| Linux `--app` | `…/usr/bin/<name>` | `…/usr/share/<name>` |

The result is `<base>/<relative>`, an absolute path with no `..` segments, so it
opens with `fs::open` regardless of the working directory — including a macOS
`.app` launched from Finder or a mounted `.AppImage`. Resolution reads only the
executable's own path (`/proc/self/exe` on Linux, `_NSGetExecutablePath` on
macOS); it never consults `$APPDIR` or any other environment variable.

Two failures are catchable with `TRAP`. When the executable path cannot be
acquired (the same condition `os::executablePath` handles) the call raises
`ErrUnsupported` (`77050007`). When `relative` contains a `.` or `..` **path
component** it raises `ErrInvalidPath` (`77030002`) — a resource path must not
navigate out of the base. A dot *inside* a filename (`song.ogg`, `..foo`,
`a..b`) is fine; only a whole component that is exactly `.` or `..` is rejected.
A leading `/` in `relative` is left as-is (it collapses harmlessly under the
base). [[src/target/shared/code/os/paths.rs:lower_resource_path]]

## The OS seam

Each `os` call — except the two build-target constants — lowers to a small
runtime helper that wraps one or two libc primitives and returns through the
standard four-register fallible result form (an infallible call still sets the OK
tag and puts its value in the result register). The seam is entirely libc:
`getenv` (reads and presence), `setenv`/`unsetenv` (mutation), the platform
`environ` array accessor (snapshot), `getpid`, `sysconf`, `gethostname`,
`getuid`/`getpwuid`, and the platform executable-path primitive (shared by
`os::executablePath` and `os::resourcePath`). `os::name` and
`os::arch` bypass the seam entirely — they are constants chosen from the target
triple at build time. `os::args` reads the entry-captured argument globals rather
than a syscall. Every result — `String`, `Boolean`, `Integer`, `List`, `Map` —
is built directly in the arena as an owned value, so nothing here holds a resource
handle or needs a `CLOSE`. [[src/target/shared/code/os/mod.rs:lower_os_helper]]

## Error model

| Code | Name | Raised by |
| --- | --- | --- |
| `77050004` | `ErrNotFound` | `os::getEnv` on an unset variable |
| `77050002` | `ErrInvalidArgument` | `os::setEnv` on an empty name or a name containing `=` |
| `77050007` | `ErrUnsupported` | `os::hostName` / `os::userName` / `os::executablePath` / `os::resourcePath` when the host lookup fails |
| `77030002` | `ErrInvalidPath` | `os::resourcePath` when `relative` contains a `.` or `..` path component |
| `77010001` | `ErrOutOfMemory` | any call, when the arena cannot allocate the returned value (or `os::setEnv` its stored variable) |

`os::getEnvOr`, `os::hasEnv`, `os::unsetEnv`, `os::environ`, `os::args`,
`os::name`, `os::arch`, `os::pid`, and `os::cpuCount` have no failure mode of
their own beyond a possible `ErrOutOfMemory` on the allocating ones
(`os::getEnvOr`, `os::environ`, `os::args`); `os::hasEnv`, `os::unsetEnv`,
`os::name`, `os::arch`, `os::pid`, and `os::cpuCount` never raise at all. A raised
error propagates like any fallible call and can be caught with `TRAP`.
