# MFBASIC for Go developers

MFBASIC compared with Go: message passing you can't opt out of, error propagation you don't write

## Introduction

Go is MFBASIC's closest philosophical neighbor: values over objects, plain
structs, explicit composition over inheritance, native binaries, concurrency
by message passing. If Go's design goals resonate with you, MFBASIC will feel
like the same goals pushed one notch further — the places where Go trusts
convention ("share memory by communicating", "check your errors", "don't
forget the defer") are the places MFBASIC compiles the convention in.

Three headline differences. Goroutines *can* share memory, so the race
detector exists; MFBASIC threads cannot. Go errors are values you check and
return by hand at every call; MFBASIC errors are values whose propagation is
automatic. And Go has no destructor story beyond `defer`; MFBASIC ties
cleanup to lexical scope, with a compile-time guarantee. There is also no GC:
every value has one owner and drops deterministically at scope exit.

## Ownership and threading

"Share memory by communicating" is Go's advice; goroutines still share the
address space, so a captured pointer or a map touched from two goroutines is
a race the compiler happily builds. You reach for channels *and* the race
detector *and* `sync.Mutex`, depending on the day.

```go
go func() { total += countWords(line) }()   // races with the loop; builds fine
```

MFBASIC has one concurrency shape, and it is the one Go recommends: an
isolated worker plus typed queues. Workers share nothing — no package
variables, no captured closures over your locals, no maps in common.

```
' text_workers package
IMPORT strings
IMPORT thread

EXPORT ISOLATED FUNC wordCount(w AS ThreadWorker OF String TO Integer, seed AS String) AS Integer
  MUT total = 0
  MUT line AS String = thread::receive(w)    ' block until the parent sends
  WHILE line <> "done"
    total = total + len(strings::split(line, " "))
    line = thread::receive(w)
  END WHILE
  RETURN total
END FUNC

' main package
IMPORT text_workers
IMPORT thread

LET t = thread::start(text_workers::wordCount, "")
thread::send(t, "the quick brown fox")
thread::send(t, "jumps over the lazy dog")
thread::send(t, "done")
LET words = thread::waitFor(t)      ' 9 — the worker's result, or its Error
```

The queues are bounded like a buffered channel, and blocking sends/receives
carry timeouts. What's new coming from Go: `send` *moves* the value — the
receiver owns its copy outright, so there is no aliased slice backing array
on the other side. And the worker's result comes back through `waitFor` as a
typed value-or-`Error` — the `result, err := <-resultCh` plumbing plus
`sync.WaitGroup` bookkeeping, collapsed into the thread handle itself.

## Error handling

You already believe errors are values. The cost in Go is that every value
must be moved by hand:

```go
text, err := os.ReadFile(path)
if err != nil {
    return 0, fmt.Errorf("config %s: %w", path, err)
}
```

— three lines per call, and forgetting the check is legal. MFBASIC keeps
errors as values (a record with `code`, `message`, `source`) and makes the
`if err != nil { return err }` the language's default: a failing call
transfers control to the function's `TRAP`, or fails the function to its
caller with the same error. You write the handler once, not per call.

```
IMPORT errorCode
IMPORT fs
IMPORT strings

FUNC loadPort(path AS String) AS Integer
  LET text = fs::readText(path)             ' may fail — auto-propagates to the TRAP
  LET port = toInt(strings::trim(text))     ' may fail — same
  IF port < 1 OR port > 65535 THEN FAIL error(77050002, "port out of range")
  RETURN port

  TRAP(err)
    IF err.code = errorCode::ErrPathNotFound THEN RETURN 8080     ' missing file: use the default
    FAIL error(err.code, "config " & path & ": " & err.message)   ' anything else: add context
  END TRAP
END FUNC

' handle one call locally instead of writing a handler:
LET port = loadPort("port.conf") TRAP(e)
  RECOVER 8080
END TRAP
```

The mapping is direct: `err.code = errorCode::ErrPathNotFound` is your
`errors.Is(err, fs.ErrNotExist)`; `FAIL error(err.code, "config … " &
err.message)` is your `fmt.Errorf("…: %w", err)` — and `err.source` keeps the
original file/line/column the way `%w` keeps the chain. The inline
`TRAP`/`RECOVER` is the "use a default on error" pattern without the
temporary `err` variable. There is no `panic`/`recover` split: one model
covers everything, and it cannot be ignored the way an unchecked `err` can.

## Data vs behavior

Go structs with methods become MFBASIC records without them — behavior lives
in package functions. You lose the receiver syntax; you keep everything you
actually use it for, since there is no inheritance in either language.

```
TYPE Invoice
  id    AS Integer
  total AS Float
  paid  AS Boolean
END TYPE

FUNC markPaid(inv AS Invoice) AS Invoice
  RETURN WITH inv { paid := TRUE }          ' a new value; the argument is untouched
END FUNC

FUNC totalDue(invoices AS List OF Invoice) AS Float
  MUT due = 0.0
  FOR EACH inv IN invoices
    IF NOT inv.paid THEN due = due + inv.total
  NEXT
  RETURN due
END FUNC
```

Bindings are immutable by default (`LET`), so `markPaid` returning an updated
copy is the norm rather than a pointer-vs-value receiver decision. Where Go
models a closed set of variants with an interface plus a type switch —
relying on you to keep the switch complete — MFBASIC has closed unions with
compile-checked exhaustiveness:

```
TYPE Card
  amount AS Float
END TYPE

TYPE Wire
  amount AS Float
END TYPE

UNION Payment                               ' a closed sum — no open interface
  Card
  Wire
END UNION

FUNC fee(p AS Payment) AS Float
  MATCH p                                   ' compile error if a member is missed
    CASE Card(c)
      RETURN c.amount * 0.029

    CASE Wire(w)
      RETURN 0.5
  END MATCH
END FUNC
```

`MATCH` is the type switch where `default:` is never silently required and a
new member breaks every switch at compile time — the sealed-sum semantics Go
programs approximate with unexported-method interfaces.

## Deterministic resource cleanup

`defer f.Close()` is good — when it's there, when it's after the error check,
and when function scope is the scope you wanted (deferring in a loop leaks
until return). MFBASIC attaches the close to the binding instead of to a
statement you remember to write:

```
IMPORT fs

FUNC copyHeader(src AS String, dst AS String) AS Integer
  RES input  = fs::openFile(src)            ' owned by this scope
  RES output = fs::open(dst, "write")
  MUT copied = 0
  WHILE copied < 10 AND NOT fs::eof(input)
    fs::writeAll(output, fs::readLine(input) & "\n")   ' a failure here still closes both
    copied = copied + 1
  END WHILE
  RETURN copied              ' output closes, then input — reverse declaration order
END FUNC
```

`RES` is a `defer Close()` you cannot forget, scoped to the block rather than
the function, running on every exit path — return, `FAIL`, propagated error —
in reverse declaration order, just like stacked defers. The compiler tracks
the close obligation: use-after-close and double-close are build failures,
not runtime surprises. Want the close error `defer` swallows? Call
`fs::close(f)` explicitly and it propagates like any other call.

## Checked numeric semantics

Go wraps integer overflow silently, and narrowing conversions of runtime
values wrap the same way (`int8(n)` where `n` is 300 gives 44 — only constants
are checked). NaN flows through float code until it reaches your output.
Division by zero panics for ints and yields `+Inf` for floats. MFBASIC replaces all of those outcomes with one: a defined failure in
the standard error model.

```
FUNC nextId(id AS Integer) AS Integer
  RETURN id + 1                 ' at the 64-bit max this FAILs — it never wraps
END FUNC

FUNC percentDone(done AS Integer, total AS Integer) AS Integer
  ' total = 0 makes the DIV produce an infinity; it is caught the moment it is
  ' observed, and percentDone fails instead of returning garbage.
  RETURN toInt(toFloat(done) DIV toFloat(total) * 100.0)
END FUNC

LET b = toByte(300)             ' fails: ErrOverflow — Byte is 0..255
LET n = toInt("12,5")           ' fails: ErrInvalidFormat — no partial parses
```

Integer overflow fails with `ErrOverflow` (77050010) — never wraps — and
because it is an ordinary error, a `TRAP` can handle it instead of a
process-killing panic. `Float` is IEEE binary64, but no NaN or infinity can
ever be stored, returned, or printed: it fails at the observation boundary.
Conversions produce the exact value or fail — there is no silent truncation
anywhere in the numeric tower.
