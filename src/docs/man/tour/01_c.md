# MFBASIC for C developers

MFBASIC compared with C: ownership without malloc, errors without errno

## Introduction

MFBASIC compiles to a native executable, like C — no VM, no interpreter, no
garbage collector, and math kernels that don't even link libm. The mental
model you already have mostly carries over: plain data in structs, behavior in
free functions, one owner per allocation, deterministic teardown.

What changes is who enforces it. The single-owner discipline you maintain by
convention — this function allocates, that one frees, don't use it after — is
the type system here. There is no `malloc`/`free` pair to balance: every value
has exactly one owning binding and is dropped when its scope exits. There are
no pointers in the language, so there is nothing to dangle, alias, or
double-free; those bugs are compile errors, not Tuesday. And there is no
undefined behavior: signed overflow, division by zero, and bad conversions are
all defined to fail with an error value you can handle.

## Ownership and threading

In C, threads share one address space, and correctness is a discipline: every
byte touched by two threads needs a mutex, an atomic, or a happens-before
argument, and the compiler verifies none of it.

```c
pthread_mutex_lock(&m);      /* forget this once and it still compiles */
total += count_words(line);
pthread_mutex_unlock(&m);
```

MFBASIC threads share nothing — no globals, no heap, no mutable state. A
worker is an `ISOLATED FUNC` exported from a package; it communicates over
bounded, typed queues. Data races are not detected — they are unrepresentable.

```
' text_workers package
IMPORT strings
IMPORT thread

EXPORT ISOLATED FUNC wordCount(w AS ThreadWorker OF String TO Integer, seed AS String) AS Integer
  MUT total = 0
  MUT line AS String = thread::receive(w)    ' block until the parent sends
  WHILE line <> "done"
    total = total + len(strings::split(line, " "))
    line = thread::receive(w)
  WEND
  RETURN total
END FUNC

' main package
IMPORT text_workers
IMPORT thread

LET t = thread::start(text_workers::wordCount, "")
thread::send(t, "the quick brown fox")
thread::send(t, "jumps over the lazy dog")
thread::send(t, "done")
LET words = thread::waitFor(t)      ' 9 — the worker's result, or its Error
```

A message is owned by exactly one side at a time: `send` moves the value into
the queue, `receive` moves it out. There is no shared buffer to lock because
there is no shared buffer. `waitFor` collects the worker's return value the
way `pthread_join` collects a `void *` — except it is typed, and a worker that
failed delivers its `Error` instead of a value you must remember to check.

## Error handling

C reports failure through return values and `errno`, and nothing makes you
look: an unchecked `fopen` is a crash three calls later. MFBASIC keeps the
"errors are values" model — an `Error` is a plain record with `code`,
`message`, and `source` — but wires the checking into the language. Every
call either produces its value or fails; on failure, control transfers to the
enclosing `TRAP`, or the function fails to its caller with the same `Error`.
The propagation you write by hand in C (`if (rc != 0) return rc;` after every
call) is the default behavior of calling a function.

```
IMPORT errorCode
IMPORT fs
IMPORT strings

FUNC loadPort(path AS String) AS Integer
  LET text = fs::readText(path)             ' may fail — auto-propagates to the TRAP
  LET port = toInt(strings::trim(text))     ' may fail — same
  IF port < 1 OR port > 65535 THEN FAIL error(77050002, "port out of range")
  RETURN port

  TRAP(err)
    IF err.code = errorCode::ErrPathNotFound THEN RETURN 8080     ' missing file: use the default
    FAIL error(err.code, "config " & path & ": " & err.message)   ' anything else: add context
  END TRAP
END FUNC

' handle one call locally instead of writing a handler:
LET port = loadPort("port.conf") TRAP(e)
  RECOVER 8080
END TRAP
```

The happy path reads like the C you wish you could write — no `rc` variables,
no `goto fail` — while the failure path is one handler per function, at the
bottom, like a cleanup label that receives a real value instead of whatever
`errno` holds by the time you read it. `err.source` carries the file, line,
and column where the error was created, so the diagnostic you'd reconstruct
from a core dump is already in the value. There are no exceptions and no
`longjmp`: propagation is ordinary value flow, visible to the auditing tools.

## Data vs behavior

This is the C model, kept: data is a struct, behavior is a function that takes
one. MFBASIC never asks you to unlearn it — there are no methods to attach, no
`self`, no vtables.

```
TYPE Invoice
  id    AS Integer
  total AS Float
  paid  AS Boolean
END TYPE

FUNC markPaid(inv AS Invoice) AS Invoice
  RETURN WITH inv { paid := TRUE }          ' a new value; the argument is untouched
END FUNC

FUNC totalDue(invoices AS List OF Invoice) AS Float
  MUT due = 0.0
  FOR EACH inv IN invoices
    IF NOT inv.paid THEN due = due + inv.total
  NEXT
  RETURN due
END FUNC
```

The differences are the checks. A constructor must initialize every field —
there is no uninitialized struct memory. `WITH` is designated-initializer
update as an expression, producing a new value. And where a C `union` is an
unchecked reinterpretation of bytes, an MFBASIC `UNION` is a tagged sum the
compiler polices:

```
TYPE Card
  amount AS Float
END TYPE

TYPE Wire
  amount AS Float
END TYPE

UNION Payment                               ' a closed, tagged sum
  Card
  Wire
END UNION

FUNC fee(p AS Payment) AS Float
  MATCH p                                   ' compile error if a member is missed
    CASE Card(c)
      RETURN c.amount * 0.029

    CASE Wire(w)
      RETURN 0.5
  END MATCH
END FUNC
```

`MATCH` reads the tag for you and binds the payload with its correct type —
the tag-plus-`switch` pattern you write by hand, except reading the wrong
member is impossible and a missing `case` fails the build instead of falling
through.

## Deterministic resource cleanup

The C idiom for multi-resource functions is the `goto cleanup` ladder: every
early return must pass the labels in the right order, and every new resource
means auditing every exit path. MFBASIC's `RES` binding is that ladder,
generated and verified:

```
IMPORT fs

FUNC copyHeader(src AS String, dst AS String) AS Integer
  RES input  = fs::openFile(src)            ' owned by this scope
  RES output = fs::open(dst, "write")
  MUT copied = 0
  WHILE copied < 10 AND NOT fs::eof(input)
    fs::writeAll(output, fs::readLine(input) & "\n")   ' a failure here still closes both
    copied = copied + 1
  WEND
  RETURN copied              ' output closes, then input — reverse declaration order
END FUNC
```

Both handles close on every exit path — normal return, an early `FAIL`, or a
propagated error — in reverse declaration order, exactly like a well-ordered
cleanup ladder. Using a handle after closing it, or closing it twice, is a
compile error (the move analysis tracks the close obligation the way you track
it in your head). To observe a close failure — the `fclose` return code
everyone ignores — call `fs::close(f)` explicitly and it propagates like any
other call.

## Checked numeric semantics

In C, `INT_MAX + 1` is undefined behavior, `x / 0` is undefined behavior, and
a narrowing conversion silently drops bits. Every MFBASIC arithmetic edge is
defined — not saturating, not wrapping, but *failing*, with an ordinary
`Error` that follows the propagation rules above:

```
FUNC nextId(id AS Integer) AS Integer
  RETURN id + 1                 ' at the 64-bit max this FAILs — it never wraps
END FUNC

FUNC percentDone(done AS Integer, total AS Integer) AS Integer
  ' total = 0 makes the DIV produce an infinity; it is caught the moment it is
  ' observed, and percentDone fails instead of returning garbage.
  RETURN toInt(toFloat(done) DIV toFloat(total) * 100.0)
END FUNC

LET b = toByte(300)             ' fails: ErrOverflow — Byte is 0..255
LET n = toInt("12,5")           ' fails: ErrInvalidFormat — no partial parses
```

`Integer` is checked 64-bit arithmetic: overflow fails with `ErrOverflow`
(77050010) on every target, with no `-fwrapv`-shaped flags to negotiate.
`Float` is IEEE binary64, but a NaN or infinity can never reach a variable,
field, or return value — it fails at the boundary where it would become
observable. Conversions are total functions with defined failure, so the
sanitizers you'd deploy to find these bugs have nothing left to find.
