NAME
    lambda - anonymous function values, function types, and capture rules

SYNOPSIS
    LAMBDA(params) -> expression
    LAMBDA(name = expression)            ' non-escaping callback mutation form

DESCRIPTION
    A lambda is an anonymous function value. It is written `LAMBDA`, a
    parenthesized parameter list, `->`, and a single body expression:

        LAMBDA(value AS Integer) -> value + 1

    Each parameter must declare an `AS` type; parameter types are not
    inferred and parameters cannot declare default values. The result type is
    inferred from the body expression. A lambda whose body is an assignment
    `name = <expression>` (see CAPTURE RULES) instead mutates `name` and has
    result type `Nothing`.

    Lambdas are ordinary values. They have the function type
    `FUNC(argTypes) AS ReturnType` and can be:

        - bound to a `LET` of function type
        - passed to higher-order built-ins (`forEach`, `transform`, `filter`,
          `reduce`, ...)
        - returned from a function
        - stored in records and collections

    all subject to the capture rules below.

FUNCTION TYPES
    The type of a lambda is `FUNC(argTypes) AS ReturnType`. A binding of
    function type holds a lambda (or a named function used as a value) and is
    called like any function:

        LET addOne AS FUNC(Integer) AS Integer = LAMBDA(x AS Integer) -> x + 1
        LET two AS Integer = addOne(1) + 1

    A function with no parameters or no result is written `FUNC() AS T` and
    `FUNC(T) AS Nothing` respectively.

CAPTURE RULES
    A lambda may reference bindings from the enclosing scope; these are its
    captures. Capture is by value: the closure deep-copies each captured
    binding into an independent copy that outlives the capturing scope, so a
    captured value never observes the original binding's later mutations.

        - Copyable `LET` bindings may be captured by value.
        - Capturing a `MUT` binding is rejected everywhere except in a
          compiler-proven non-escaping callback position (see below).
        - Capturing a resource (`RES`) or any other non-copyable value is
          always rejected.

    Each rejected capture is reported as `TYPE_LAMBDA_CAPTURE_UNSUPPORTED`.

  Non-escaping MUT capture
    A small set of built-ins take a callback in a position the compiler knows
    to be non-escaping: the callback runs only synchronously during the call,
    is never stored, returned, forwarded, or run from another thread. Today
    `forEach` is the only such position.

    A lambda passed there may capture an outer `MUT` binding and mutate it.
    The binding is loaned to the callback for the duration of the call â€” a
    borrow of the live binding, not a copy â€” and is the outer binding's again
    once the call returns. Use the assignment body form to update it:

        MUT total AS Integer = 0
        forEach(items, LAMBDA(x AS Integer) -> total = total + x)
        ' total now holds the sum

    Mutating a captured `MUT List` or `MUT Map` works the same way; a grow or
    in-place update is reflected in the outer binding after the call. While
    the callback is in flight the outer binding is logically unavailable
    except through the capture; it is the outer binding's again once `forEach`
    returns. Capturing a resource through such a callback is still rejected.

WHAT IS NOT POSSIBLE
    - Capturing a `MUT` binding in an escaping closure â€” assigning it to a
      `LET`, returning it, storing it in a record/collection, sending it to a
      thread, or passing it to an unknown function â€” is rejected. Only the
      proven non-escaping callback positions above are exempt.
    - Observing a captured value's later mutations through a by-value capture:
      the closure holds an independent copy.
    - Capturing resources or other non-copyable values.

EXAMPLES
    Allowed â€” accumulate into an outer `MUT` through `forEach`:

        MUT total AS Integer = 0
        forEach(items, LAMBDA(x AS Integer) -> total = total + x)

    Rejected â€” a `MUT`-capturing lambda that escapes its scope:

        FUNC makeCounter() AS FUNC() AS Integer
          MUT total AS Integer = 0
          RETURN LAMBDA() -> total = total + 1   ' escapes: rejected
        END FUNC

    The returned closure would outlive `total`, so the capture cannot be a
    call-bound loan and is rejected as `TYPE_LAMBDA_CAPTURE_UNSUPPORTED`.
